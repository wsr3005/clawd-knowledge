# AI / Agent 战略情报｜2026-07-17

**一句话结论：** 今天真正改变动作的是两条基础设施信号和一条安全红线：Codex 全权限模式已出现误删 `$HOME` 的已确认案例，Grok Build 则把完整 Agent harness 开源，Codex 的 PR 审查—修改闭环也正式进入产品界面。

## 1. Codex / GPT-5.6 的全权限误删风险已被团队确认

【事实】OpenAI Codex 的 Tibo Sottiaux 表示，团队调查了少量 GPT-5.6 意外删文件的报告；最常见链路是：启用 Full Access、未使用 sandbox/Auto Review，模型试图把 `$HOME` 改作临时目录，随后误删真实 `$HOME`。团队正在修改 developer message、引导使用更安全的权限模式并增加 harness 防护；详细复盘尚未发布。其“极少发生”是团队表述，目前没有公开发生率。

【影响】这不是“模型偶尔犯错”的抽象风险，而是权限配置会把一次变量错误放大成用户目录级损失。昱桥的 Codex/Hermes 自动化会操作真实仓库和本机文件，默认 Full Access + 无审核不再是可接受的常态配置。

【动作】立即把安全基线定为：默认 sandbox/工作区写权限；Full Access 仅用于明确必要的短任务；禁止把 `$HOME` 作为临时目录目标；删除、覆盖、批量移动前要求 diff/清单与人工确认；关键仓库保持远端或独立备份。等官方 post-mortem 发布后，再核对是否需要新增 hook 或路径级 deny rule。

【证据】[Tibo Sottiaux：文件删除调查与缓解措施](https://x.com/thsottiaux/status/2077630111499882637)

【可信度/待核验点】高：Codex 团队成员直接说明调查结论。待核验：官方详细复盘、影响版本范围、实际发生率与最终 harness 修复是否已全面上线。

## 2. xAI 开源 Grok Build：Agent harness 已成为可直接拆解的公共基础设施

【事实】xAI 官方宣布 Grok Build 的 CLI/TUI 与 Agent runtime 已开源，仓库包含上下文组装、工具实现、文件/VCS/执行/checkpoint、sandbox，以及 Skills、Plugins、Hooks、MCP、headless 和 ACP 等路径；第一方代码采用 Apache-2.0，可从源码构建并连接本地推理。官方仓库同时注明：代码从内部 monorepo 定期同步，当前不接受外部贡献。

【影响】对 Hermes/OpenClaw 的价值不在于换一个 coding agent，而在于出现了一个可审计、可复用的生产级 harness 对照样本。尤其值得横向比对 context assembly、tool dispatch、checkpoint、sandbox、headless/ACP 与 Skills/Plugins 的装载边界，减少自己凭感觉造轮子。

【动作】列入近期架构拆解对象，但不直接迁移：优先审计五块——上下文拼装、权限/沙箱、长任务生命周期、checkpoint/恢复、Skills/Plugins/Hooks 装载；只提炼能改善 Hermes/OpenClaw 的机制，并保留许可证与第三方 notice。仓库是单次公开快照/周期同步且不收 PR，不应把它误判为开放治理的上游依赖。

【证据】[xAI 官方公告](https://x.ai/news/grok-build-open-source) · [GitHub：xai-org/grok-build](https://github.com/xai-org/grok-build) · [AI HOT 聚合入口（6 个独立信源）](https://aihot.virxact.com/items/cmrmkn6j703yfbiul4fa0bai9)

【可信度/待核验点】高：官方公告、源码仓库与许可证均可访问。待核验：后续同步频率、与内部生产版本的差异、local inference 的兼容范围。

## 3. Codex 把 PR 问答、审查反馈与行内修补收进同一闭环

【事实】OpenAI Developers 官方宣布：Codex 的 PR Chat 可针对特定 Pull Request 在上下文中问答；Inline code editing 可把审查意见交给 Codex，查看 proposed patch，并在界面内编辑、接受或拒绝。官方帖未说明套餐、平台与全量开放范围。

【影响】这会压缩“Agent 提交 → 人工审查 → 反馈 → Agent 修改 → 再审”的上下文切换成本。对多 Agent/Kanban 流程而言，PR 可以更明确地成为可验证交付边界，而不是把长期主线程继续拖长。

【动作】若当前 Codex 账户已出现该能力，就在下一个真实 PR 试跑：要求每个 ticket 独立 PR、PR Chat 只处理该切片、所有 follow-up patch 仍须跑测试并由人确认后合并；不要因为界面闭环而跳过 CI、diff 审查或权限边界。

【证据】[OpenAI Developers 官方发布](https://x.com/OpenAIDevs/status/2077902662973190570)

【可信度/待核验点】高：第一方产品账号直接发布。待核验：具体可用平台、套餐、地区及 rollout 节奏。
